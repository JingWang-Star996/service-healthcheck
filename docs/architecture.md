# 架构设计 — 为什么用脚本而不是 AI agent

## 问题

OpenClaw 等 AI Agent 框架的自检/健康检查，常见做法是：
1. 定时唤醒 agent
2. Agent 手动执行 shell 命令（ps、ss、find 等）
3. Agent 分析输出，判断是否异常
4. Agent 决定是否通知用户

**问题**：
- 每次消耗 ~2K tokens（模型推理 + 工具调用）
- 一小时一次 = 每天 48K tokens，月消耗 ~1.5M tokens
- 判断逻辑不透明，每次可能不同
- 无法积累历史趋势（没有结构化存储）

## 解决方案

**自检逻辑下沉到脚本，AI 只读结果**。

```
┌─────────────────────┐     ┌─────────────────────┐
│  系统 crontab        │     │  OpenClaw cron       │
│  (每小时整点)        │     │  (错开几分钟)        │
│                     │     │                     │
│  bash healthcheck   │────▶│  read status.json    │
│  --json             │     │                     │
│                     │     │  ok → NO_REPLY       │
│  Token: 0           │     │  ≠ok → notify user   │
└─────────────────────┘     │  Token: ~50          │
                            └─────────────────────┘
```

## 为什么有效

### 1. 自检是确定性问题，不需要 AI

- "进程是否存在？" → `pgrep`
- "内存是否超限？" → 数值比较
- "最近有无崩溃？" → `grep` 日志

这些是 **确定性判断**，正则/阈值就够了，不需要 LLM 推理。

### 2. AI 的价值在 "处理异常"，不在 "发现异常"

- 脚本负责发现异常（快速、准确、零成本）
- AI 负责解释异常、建议修复、执行修复（这才是 LLM 擅长的）

### 3. 结构化输出 = 趋势分析的基础

脚本输出 JSON，自然形成时间序列数据：

```json
{"ts":"2026-05-01T11:00:00","check":"L0-Q2","metric":"status","value":1286,"status":"ok"}
{"ts":"2026-05-01T12:00:00","check":"L0-Q2","metric":"status","value":1312,"status":"ok"}
{"ts":"2026-05-01T13:00:00","check":"L0-Q2","metric":"status","value":1350,"status":"ok"}
```

有了这些数据，可以做：
- 内存泄漏检测（线性回归斜率）
- 趋势预警（连续 N 次上升）
- 历史回溯（某时间点发生了什么）

### 4. 退出码 = 零成本状态传递

```bash
bash healthcheck.sh
echo $?  # 0=ok, 1=warn, 2=critical
```

任何上层系统（cron、CI/CD、监控面板）都可以通过退出码快速判断状态。

## 适用场景

| 场景 | 适合脚本 | 适合 AI agent |
|------|---------|--------------|
| 进程/内存/磁盘检查 | ✅ 确定性判断 | ❌ 浪费 token |
| 日志关键词匹配 | ✅ grep 就够了 | ❌ 浪费 token |
| 阈值比较 | ✅ 数值运算 | ❌ 浪费 token |
| 异常根因分析 | ❌ 需要推理 | ✅ LLM 擅长 |
| 修复建议 | ❌ 需要上下文 | ✅ LLM 擅长 |
| 跨系统协调 | ❌ 需要 API 调用 | ✅ Agent 可以 |

## 推广到其他场景

这套思路可以推广到任何 "定期检查 + 异常通知" 的场景：

1. **CI/CD 健康检查**：脚本检查构建状态、测试覆盖率、依赖过期
2. **数据库巡检**：脚本检查慢查询、连接池、磁盘空间
3. **K8s 集群巡检**：脚本检查 Pod 状态、资源使用、事件日志
4. **API 可用性监控**：脚本检查端点响应时间、错误率

**核心原则**：确定性检查用脚本，不确定性问题用 AI。
